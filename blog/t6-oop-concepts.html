Inheritance and Composition

An intro to understanding inheritance

2015-06-04

Inheritance

In my last post, I talked about Ruby classes and instance methods. This week, I'll take a step back from a class, and look at the idea of inheritance with a class's superclass and subclass. A class is itself an instance of another class, if it's not a defined class, then it's an instance of the class Object. Fun right? The Class class is also made up of its own methods, which any class created from Class has access to, in a kind of heirarchy. This kind of access is called Inheritance, because you can think of Class as a parent, and each proceeding subclass as its children, inheriting all of its traits (i.e. methods). Although a child inherits traits from two parents, in Ruby a class can only inherit methods from one class. This isn't true across programming languages, since some languages allow multiple inheritances from multiple classes.

This example shows a common notation for declaring a super-class (parent class) for a subclass (child class) by using the < symbol. The < symbol in this example is like saying "Coffee inherited from Beverage".

          class Beverage
            def drink
              puts "You are hydrated"
            end
          end

          class Coffee < Beverage
            def craft
              puts "You're an excellent barista"
            end
          end

        cappuccino = Coffee.new => New instance of the class Coffee
        cappuccino.drink => "You are hydrated"
        cappuccino.craft => "You're an excellent barista"

This is called Implicit Inheritance, since the class Coffee implicitly inherits all methods from the Beverage class. Even though we didn't define the drink method within the Coffee class, any new object of the Coffee class can receive the drink method, because it inherited it from Beverage.

An interesting point with inheritance is that a class can override inherited methods. To do this, a subclass will change a superclass's method by rewriting it in their own class. This means that every new class can be redefined based on the original blueprints given from a superclass, or as Sandi Metz puts it in Practical Object Oriented Design in Ruby: "Subclasses are thus specializations of their superclasses."

So where does the class Class come from if everything is inherited from a superclass? The very root of it all comes from the BasicObject class--the parent class of all Ruby classes. This BasicObject class has very few methods.

The next step is to understand when and how to use inheritance. We know WHAT inheritance is, it's time to look at practical examples of WHEN to use it appropriately. For now, I think you would use inheritance when you build a class that has obvious signs of multiple categories in 1 class. This might tell you that you have 1 overarching category, and multiple subcategories(or subclasses!). I'll be learning more about this in the coming weeks, and I hope to expand on this blog about when I have more practical experience using Inheritance.

Composition

A great way to think about the difference between inheritance and composition, is to use the Learn Ruby the Hard Way sentences "is-a" and "has-a". In the earlier example, the Coffee class inherited from the Beverage class, so we can say: Coffee is-a Beverage.

So what does it mean if composition is a "has-a" phrase? Instead of creating a subclass from a superclass, composition means there is one class that is using the methods from another class. So in this case, Coffee has-a Beverage in its class definition. This allows the class Coffee to use methods from the Beverage class as it needs to, but to also define its own methods and avoid having to rewrite any of the Beverage methods.

          class Beverage
            def drink
              puts "You are hydrated"
            end
          end

          class Coffee
            def initialize
              @beverage = Beverage.new
            end

            def craft
              puts "You're an excellent barista"
            end

            def drink 
              @beverage.drink
            end

          end

        cappuccino = Coffee.new => New instance of the class Coffee and sets the variable @beverage to a new instance of the Beverage class
        cappuccino.drink => "You are hydrated"
        cappuccino.craft => "You're an excellent barista"

Here you see that the Coffee class instantiates itself by setting an instance variable @beverage to a new instance of the Beverage class. When the Coffee object cappuccino receives the drink method, it's now referencing the new instance of the Beverage class that is calling the drink method from within the Beverage class. Here we see that instances of the Coffee class can use methods from the Beverage class without being inherited, and instead by using an instance of the Beverage class within its own methods. We can say that the Coffee class "has a" method from an instance of the Beverage class.

Inheritance vs Composition

So when would you use inheritance or composition? What are the various situations one is preferable over another? This is something I'm still trying to figure out, and I expect an understanding will come with practice. Based on the resources below, it seems you'd want to use composition to package modules that you can resuse in many different situations. Inheritance seems to be useful when conceptually you have a class that naturally has subclasses, so that you can keep it clear in your mind what class you need at what time and how they interact.
