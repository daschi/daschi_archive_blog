Issues in Tech: Localization

Localizing software for language and culture

2015-05-24

Localization and Internationalization

This week in DevBootcamp, we are asked to select an issue in the tech industry to write about. Instinctually my mind wandered to issues like women in tech, diversity in tech, educational opportunities for the low-income students to be exposed to tech, etc. Don't get me wrong, I could go on and on for hours on those topics, which is why I've decided to talk about something I've been marginally involved with in the past year, but don't particularly know too many details about.

That's why I'll try to focus this post on localization and internationalization of software and applications. I know a bit because I've been a localization project manager for the past year, which means I worked with various (tech company) clients to localize and internationalize their software, web/mobile apps, and sites into whichever languages are needed for their market. I've heard many times from engineers in the field that it's particularly difficult to write a program that will later be internationalized. My quest for today is to find out some reasons why it is difficult to internationalize code.

What are Localization and Internationalization?

Localization: L10n

Simply, localization adapts a product, application or content to match the linguistic and cultural needs of a specific audience. This is done by translators, usually with a CAT (Computer-Assisted Translation) Tool. CAT tools work by segmenting sentences into either whole sentences or phrases where it seems semantically logical to segment. This helps the translator break up the meaning of a sentence and restructure it as they need to for their language, and also saves the segments in a database called a Translation Memory (TM). The TM stores all the previously translated segments so that files that are updated later down the road can be analyzed and leveraged using previous translations. This helps translators to recycle and reuse past translations, and it also saves time and money. Localizing a product also means testing, which is a crucial step to ensure that everything in the UI and content makes sense in context.

One more quick note on the difference between "localization" and "translation". Localizing creates a new product that is culturally and linguistically appropriate for the country and audience. For example, a file may be translated into Spanish--but which Spanish? It will need to be localized for either Spanish for Spain, for Mexico, Argentina, Guatemala, Ecuador...I think you are seeing the point now...

Internationalization: i18n

Internationalization is done by developers, and is the process of reviewing a program's files to make sure the code ready for localization.This could include pulling out content from within code (i.e. removing user interface text from Javascript), supporting local standards, such as date, time and currency, and handling Unicode or other character encoding standards. This process would ideally create a new, global product can be localized to support any locale and target country's needs.

Issues related to Internationalizing Code

So in my breakdown of internationalization and localization, I hinted at some potential issues developers face while preparing content for translation. For one, if a software or app may eventually become localized, this should be taken into account in they first planning stages of the application and site. This will save a lot of time and effort in the future to retroactively make your software localizeable.

Here are some other issues or considerations for internationalizing code that may be interesting:

  • Encoding

    • Usually some form of Unicode should be used

  • Text Direction

    • Does your UI design work for right to left languages?

  • Text spacing, buttons, windows

    • A 5 letter word in English could be a 10 letter word in German, Turkish, or French. Consider the possibility of resizing windows and buttons

  • Remove Strings into a New File

    • Do not hard code strings! Many CAT tools can't extract text from script files, so any content that needs to be translated should be pulled out and saved into a properties or resource file instead, and loaded as needed. The application can select the correct resource file for the language selected.

  • TIME

    • Time is its own issue. If you plan to localize, consider that it will take just as much time to localize (if not more) than it did to write the original source file. If you make periodic updates to source code, add in time to the development cycle for localization so that all versions are up to date around the same time frame.

  • Symbols, icons, colors

    • Do these symbols, icons, colors work for all cultures?

  • Sorting

    • Think about sorting A-Z, with all the letters in the Turkish alphabet. What about with Chinese characters?

Caveat: There is a HUGE amount of information about this, so what I've said here may or may not be accurate. END CAVEAT

I'm still learning, so although many of my "issues" sound very authoritative, think of them more as questions to open the dialogue of what we need to do to localize software.
